# Распределённый Commit Log

Прежде чем погружаться в брокеры, партиции и репликацию — нужно усвоить одну фундаментальную идею. Kafka — это не очередь сообщений. Kafka — это **распределённый commit log**. Вся её архитектура вытекает из этой единственной абстракции.

Если вы поймёте commit log — вы поймёте Kafka.

---

## Append-only log: данные добавляются только в конец

Commit log — это простейшая структура данных: файл, в который записи добавляются исключительно в конец. Ни вставки в середину. Ни изменения существующих записей. Ни удаления отдельных элементов. Только append.

Это кажется ограничением, но на самом деле это суперсила. Последовательная запись на диск — самая быстрая операция ввода-вывода, которая существует. Жёсткий диск с вращающимися пластинами даёт пропускную способность в сотни мегабайт в секунду при последовательном доступе — и деградирует до мегабайт в секунду при случайном. SSD значительно менее чувствителен к этому, но последовательный I/O всё равно быстрее. Kafka извлекает из этого максимум.

Append-only log — это та же идея, что и:

- **WAL (Write-Ahead Log)** в PostgreSQL: каждое изменение записывается в лог до применения к странице данных.
- **Binlog** в MySQL: последовательность изменений для репликации.
- **Raft log** в etcd и CockroachDB: журнал команд консенсуса.
- **Event Sourcing**: состояние системы = последовательность событий, а не текущий снапшот.

Все эти системы используют одну идею: **лог как единственный источник истины**.

## Offset: монотонный идентификатор позиции

**Offset** — это целое число. Не UUID. Не хэш. Просто целое число, начиная с 0, монотонно возрастающее в пределах партиции.

Offset решает три задачи сразу:

1. **Адресация записи.** Зная partition и offset, вы точно знаете, какую запись читать. `orders/partition=0/offset=42` — это конкретная запись.
2. **Позиция потребителя.** Consumer group хранит текущий offset (последний прочитанный + 1). При перезапуске продолжает с того же места.
3. **Replay.** Новый потребитель может начать с offset 0 и прочитать все записи с самого начала (если retention ещё не истёк).

<Callout type="note">
Offset уникален в пределах одной партиции, но не топика целиком. В топике `orders` с 6 партициями существуют одновременно offset 0 в partition 0, offset 0 в partition 1, ..., offset 0 в partition 5. Потребитель отслеживает offset **per-partition**.
</Callout>

Главное отличие от традиционных очередей: в RabbitMQ или ActiveMQ у сообщения нет постоянного адреса. После прочтения оно исчезает. В Kafka offset 42 в partition 0 существует до истечения retention — любой потребитель с правильными правами может его прочитать.

## Иммутабельность: записанное не изменяется

После того как запись добавлена в лог, она **неизменяема**. Вы не можете:

- Обновить значение записи с offset 42.
- Удалить конкретную запись (только весь сегмент по retention или compaction tombstone).
- Изменить ключ, метаданные, timestamp.

Иммутабельность — это не баг, а фича. Она даёт несколько свойств:

**Детерминированный replay.** Если consumer group A и consumer group B читают partition 0 с offset 0 независимо, они получат абсолютно одинаковую последовательность записей. Никаких гонок, никаких конфликтов.

**Простая репликация.** Фолловер-реплика копирует лог лидера через простой fetch-запрос: "дай мне всё начиная с offset N". Никакого merge, никакого conflict resolution.

**Возможность аудита.** Лог сохраняет полную историю изменений. Kafka часто используется как append-only audit log для систем с требованиями compliance.

<Callout type="warning">
Единственный механизм "удаления" данных в Kafka — это **log compaction** для топиков с `cleanup.policy=compact`. Compaction оставляет только последнее значение для каждого ключа, удаляя старые версии. Но даже здесь записи не изменяются — compaction создаёт новые сегменты с отфильтрованными записями. Подробно — в Модуле 04.
</Callout>

## Топики как категории событий

**Топик** — это именованный канал для событий определённого типа. Топик `orders` содержит события о заказах. Топик `payments` содержит события платежей. Топик `user-events` — действия пользователей.

Топик — это логическая абстракция. Физически топик разбит на **партиции**: `orders-partition-0`, `orders-partition-1`, ..., `orders-partition-5`. Каждая партиция — это отдельный append-only лог.

Зачем нужны партиции? Одна партиция = один лог = один поток. Партиции добавляют **параллелизм**:

- Несколько продюсеров пишут в разные партиции одного топика одновременно.
- Несколько потребителей в одной consumer group читают разные партиции параллельно.
- Данные партиций хранятся на разных брокерах — горизонтальное масштабирование.

<Callout type="note">
Гарантия порядка в Kafka: **строго в пределах одной партиции**. Если вам важно, чтобы события одного заказа обрабатывались по порядку, все события этого заказа должны попасть в одну партицию (через ключ сообщения: ключ `order_id` → один и тот же partition всегда).
</Callout>

## Почему не база данных?

Часто спрашивают: если Kafka хранит данные на диске, почему бы не использовать PostgreSQL?

**Kafka оптимизирована для совершенно другого сценария нагрузки:**

| Свойство | OLTP база данных | Kafka Commit Log |
|----------|-----------------|-----------------|
| Доступ к данным | Случайный (по первичному ключу, индексам) | Последовательный (offset → следующий offset) |
| Обновление | UPDATE/DELETE на конкретные строки | Только append (immutable) |
| Число читателей | Десятки concurrent connections | Миллионы consumer groups |
| Пропускная способность | Тысячи транзакций/с | Миллионы записей/с |
| Цель хранения | Текущее состояние | История изменений |
| Репликация | Сложный протокол (streaming replication, logical) | Простой: фолловер fetch от лидера |

База данных оптимизирована для **чтения текущего состояния** (SELECT WHERE id=42). Kafka оптимизирована для **потоков изменений** (дай мне всё после offset N как можно быстрее).

Именно поэтому Kafka и базы данных не заменяют, а **дополняют** друг друга. Типичная production-архитектура: OLTP-база (PostgreSQL) как источник истины текущего состояния + Kafka как commit log событий изменений (через CDC или прямую запись) + OLAP-хранилище (ClickHouse, BigQuery) как материализованный снапшот для аналитики.

## Retention: управление жизненным циклом данных

В отличие от баз данных, где данные живут "вечно" (пока не удалишь), в Kafka данные **автоматически удаляются** по истечении retention:

- `log.retention.ms` (по умолчанию: 7 дней) — удаляет сегменты старше N миллисекунд.
- `log.retention.bytes` — удаляет старые сегменты, когда суммарный размер партиции превышает порог.
- `log.retention.hours` — удобный alias для часов.

Удаление работает **посегментно**, не поозаписно. Kafka хранит лог разбитым на сегменты (по умолчанию 1 ГБ или 7 дней). Сегмент удаляется целиком, когда все его записи старше retention.

<Callout type="tip">
Для долгосрочного хранения (audit log, event sourcing) используйте `log.retention.ms=-1` (бесконечное хранение) вместе с Tiered Storage (Kafka 4.0+) — горячие данные на локальном диске брокера, холодные — в object storage (S3, GCS). Подробно в Модуле 04.
</Callout>

## Ключевые выводы

1. Kafka — это **append-only commit log**. Записи добавляются только в конец, никогда не изменяются.
2. **Offset** — монотонный целочисленный адрес записи в пределах партиции. Потребитель отслеживает offset, а не "забирает" сообщение.
3. **Иммутабельность** обеспечивает детерминированный replay, простую репликацию и возможность аудита.
4. **Топики** — логические категории, физически разбитые на партиции. Гарантия порядка — только внутри одной партиции.
5. Kafka оптимизирована для **последовательного I/O** и потоков изменений — не для случайного доступа и текущего состояния.
